۱ تست چیست و چرا اهمیت دارد؟

قبل از هر ابزار و تکنیکی، باید بدانیم اصلاً «تست» چیست و چه دردی را دوا می‌کند.

تستِ نرم‌افزار یعنی نوشتن کدی که کدِ دیگر (کدِ اصلی برنامه) را به‌طور خودکار اجرا می‌کند و بررسی می‌کند آیا رفتار آن همان چیزی است که انتظار داریم یا نه. یک تست، یک ادعا (assertion) دارد: «اگر ورودی این باشد، خروجی باید آن باشد.»

به‌جای اینکه هر بار دستی برنامه را باز کنی و چک کنی که فلان قابلیت کار می‌کند، یک بار تست می‌نویسی و از آن به بعد با یک دستور، صدها بررسی در چند ثانیه انجام می‌شود.

یک مثال ساده برای درک شهودی

فرض کن تابعی داری که دو عدد را جمع می‌کند. تست آن یعنی: «به تابع ۲ و ۳ می‌دهم، انتظار دارم ۵ برگرداند.»

example.py 
# کد اصلی
def add(a, b):
    return a + b

# تست: ادعای ما درباره رفتار تابع
def test_add_two_positive_numbers():
    assert add(2, 3) == 5

مستندسازی زنده

🎯

طراحی بهتر

⚡

بازخورد سریع

باگ را در چند ثانیه پیدا می‌کنی، نه چند روز بعد در محیط واقعی و جلوی کاربر.

😌

اعتماد و آرامش

با یک مجموعه تستِ سبز، با خیال راحت کد را منتشر می‌کنی. «کار می‌کند» دیگر حدس نیست.

💰

صرفه‌جویی

هزینه‌ی رفعِ باگ در مرحله توسعه، کسری از هزینه‌ی همان باگ در محصولِ منتشرشده است.

⚠️
باور غلط رایج: «تست وقت‌گیر است و سرعت را کم می‌کند.»

در کوتاه‌مدت کمی کندتر پیش می‌روی، اما در میان‌مدت و بلندمدت به‌شدت سریع‌تر می‌شوی؛ چون وقت را صرف دیباگِ دستی و باگ‌های تکراری نمی‌کنی و بی‌ترس تغییر می‌دهی.

۲ انواع تست و هرم تست

همه‌ی تست‌ها یک‌جور نیستند. هر نوع، سطح متفاوتی از برنامه را می‌سنجد.

نوع تستچه چیزی را می‌سنجد؟سرعتمثال در OXYGPT
واحد (Unit)یک تابع/کلاسِ کوچک، به‌صورت ایزولهخیلی سریع ⚡نرمال‌سازی نماد بازار (افزودن پیشوند FX:)
یکپارچگی (Integration)همکاری چند بخش با هممتوسط 🚶موتور تولید + منطق fallback بین دو ارائه‌دهنده
سرتاسری (E2E)کل مسیر از دید کاربرکند 🐢پیام کاربر → پاسخ نهایی بات (با mockِ تلگرام)

هرم تست (Testing Pyramid)

مارتین فاولر (Martin Fowler) توصیه می‌کند تعداد تست‌ها را مثل یک هرم توزیع کنی: پایه‌ی پهن از تست‌های واحدِ سریع، لایه‌ی میانی کمتر از تست‌های یکپارچگی، و نوکِ باریک از تست‌های سرتاسریِ کند و شکننده.

چرا هرم و نه مستطیل یا هرمِ وارونه؟

تست‌های سرتاسری کند، شکننده و پرهزینه‌اند. اگر بیشترِ تست‌هایت از این نوع باشند، مجموعه‌ات کُند و بی‌ثبات می‌شود. تست‌های واحد ارزان و سریع‌اند؛ پس بیشترِ اطمینان را از آنجا بگیر.

کالبد یک تست خوب: الگوی AAA

تقریباً هر تستِ خوب سه بخش دارد؛ به این الگو Arrange-Act-Assert (آماده‌سازی، اجرا، ادعا) می‌گویند — یک استاندارد جهانی برای خوانا و متمرکز نگه‌داشتن تست.

🔧

Arrange — آماده‌سازی

داده‌ها، اشیاء و پیش‌شرط‌ها را می‌چینی. صحنه را برای اجرا آماده می‌کنی.

▶️

Act — اجرا

دقیقاً یک رفتار را اجرا می‌کنی؛ همان تابع یا متدی که می‌خواهی بسنجی.

✅

Assert — ادعا

بررسی می‌کنی نتیجه همان چیزی است که انتظار داشتی. یک تست، یک رفتار.

test_aaa.py 
def test_paid_user_has_higher_request_limit():
    # Arrange — یک کاربر پولی و محدودیت‌هایش را آماده می‌کنیم
    user = User(tier="paid")

    # Act — سقف درخواست را می‌خوانیم
    limit = user.request_limit()

    # Assert — ادعا می‌کنیم برابر ۷۶۰ است
    assert limit == 760

۳ توسعه تست‌محور (TDD) چیست؟

TDD یعنی برعکسِ عادت رایج: اول تست را می‌نویسی، بعد کد را.

Test-Driven Development (توسعه‌ی هدایت‌شده با تست) روشی است که در آن قبل از نوشتن حتی یک خط از کدِ اصلی، اول تستی می‌نویسی که رفتار مورد نظرت را توصیف می‌کند. این تست ابتدا شکست می‌خورد (چون هنوز کدی نیست)، سپس فقط آن‌قدر کد می‌نویسی که تست سبز شود، و در پایان کد را تمیز می‌کنی.

در TDD، تست نقش «مشخصات فنی» را بازی می‌کند: تو اول تعریف می‌کنی «چه می‌خواهم»، بعد می‌روی سراغ «چطور بسازم». همین جابه‌جایی ترتیب، طراحی را به‌شکل چشمگیری بهتر می‌کند.

چرخه‌ی قرمز → سبز → بازآرایی

قلب TDD یک چرخه‌ی سه‌مرحله‌ایِ کوتاه و تکرارشونده است (Red · Green · Refactor):

🔴 قرمز · Red

یک تستِ شکست‌خورده بنویس

تستی برای رفتاری که هنوز وجود ندارد می‌نویسی. اجرا می‌کنی و می‌بینی که قرمز/شکست‌خورده است. اینجا داری تعریف می‌کنی «چه می‌خواهم بسازم».

🟢 سبز · Green

کمترین کد برای عبور

فقط آن‌قدر کد می‌نویسی که تست سبز شود — بدون وسواسِ بهینه‌سازی. هدف فعلاً «کار کردن» است، نه «زیبا بودن».

🟣 بازآرایی · Refactor

کد را تمیز کن

حالا که سبز هستی، کد را بهتر می‌کنی: حذف تکرار، نام‌های بهتر، ساختار تمیزتر — با اطمینان، چون تست‌ها اگر چیزی بشکند فوراً می‌گویند.

🔴 → 🟢 → 🟣 → 🔁 (تکرار)
✅
قانون طلایی: هرگز کدِ اصلی ننویس مگر برای عبور دادنِ یک تستِ شکست‌خورده.

و هرگز تستِ جدید ننویس تا وقتی همه‌ی تست‌های فعلی سبزند. قدم‌ها را کوچک نگه دار.

چرا مرحله «قرمز» واقعاً مهم است؟

دیدن شکستِ تست، ثابت می‌کند که تستِ تو واقعاً چیزی را می‌سنجد. تستی که هیچ‌وقت قرمز نشده، شاید اشتباه نوشته شده باشد و همیشه سبز بماند — یعنی بی‌فایده. مرحله قرمز، تستِ تو را تست می‌کند!

۴ اولین تجربه‌ی عملی TDD

بیایید یک قابلیت واقعی از OXYGPT را با TDD بسازیم: تابعی که نماد بازار را نرمال می‌کند و پیشوند صرافی را اضافه می‌کند.

🟢 قدم ۲: کمترین کد برای سبز شدن

def normalize_symbol(symbol: str) -> str:
    if ":" not in symbol:
        return f"FX:{symbol}"
    return symbol

🟣 قدم ۳: بازآرایی و افزودن حالت‌های بیشتر

حالا تست‌های بیشتری اضافه می‌کنیم (کریپتو، نماد از قبل کامل) و کد را تمیز می‌کنیم — با خیال راحت، چون شبکه‌ی ایمنیِ تست‌ها را داریم.

test_symbols.py · گسترش 
import pytest
from tools import normalize_symbol

@pytest.mark.parametrize("raw, expected", [
    ("EURUSD",        "FX:EURUSD"),
    ("FX:GBPUSD",     "FX:GBPUSD"),      # از قبل کامل → بدون تغییر
    ("BINANCE:BTCUSDT", "BINANCE:BTCUSDT"),
])
def test_symbol_normalization(raw, expected):
    assert normalize_symbol(raw) == expected

فیکسچرها (Fixtures) — آماده‌سازی قابل‌استفاده مجدد

فیکسچر، تکه‌ای از «آماده‌سازی» (بخش Arrange) است که چند تست به آن نیاز دارند؛ مثل یک دیتابیس موقت، یک کلاینت جعلی، یا داده‌ی نمونه. یک بار می‌نویسی، همه‌جا استفاده می‌کنی.

conftest.py 
import pytest
from database import DatabaseManager

@pytest.fixture
def temp_db(tmp_path):
    """یک دیتابیس SQLite موقت و تمیز برای هر تست می‌سازد."""
    db_file = tmp_path / "test.db"       # tmp_path داخلیِ pytest است
    db = DatabaseManager(str(db_file))
    yield db                            # تست اینجا اجرا می‌شود
    db.close()                          # پاک‌سازی بعد از تست (teardown)
test_database.py 
def test_new_user_starts_with_zero_requests(temp_db):
    # فیکسچر «temp_db» خودکار تزریق می‌شود
    usage = temp_db.get_usage(user_id=123)
    assert usage["requests"] == 0

def test_window_creation_is_capped_at_five(temp_db):
    for _ in range(10):
        temp_db.create_window(user_id=123)
    assert len(temp_db.list_windows(123)) == 5
✅
هر تست، دنیای خودش را دارد. چون هر تست یک temp_db تازه می‌گیرد، تست‌ها روی هم اثر نمی‌گذارند و ترتیبِ اجرا مهم نیست — یکی از ویژگی‌های کلیدی تستِ خوب (بخش I در FIRST).

پارامتری‌سازی (Parametrize) — یک تست، ده‌ها حالت

وقتی می‌خواهی یک منطق را با ورودی‌های مختلف بسنجی، لازم نیست ده تابع بنویسی. با @pytest.mark.parametrize یک تست می‌نویسی و ده‌ها حالت می‌سازی. این بهترین راه برای رسیدن به تعداد تستِ بالا با کدِ کم است.

test_limits.py 
@pytest.mark.parametrize("tier, expected_requests", [
    ("free", 25),
    ("paid", 760),
])
def test_request_limit_per_tier(tier, expected_requests):
    assert request_limit(tier) == expected_requests

# تستِ مرزها: درست زیر، دقیقاً روی، و درست بالای حد مجاز
@pytest.mark.parametrize("count, blocked", [
    (24, False),   # زیر حد → آزاد
    (25, False),   # دقیقاً روی حد → هنوز آزاد
    (26, True),    # بالای حد → مسدود
])
def test_free_tier_boundary(count, blocked):
    assert is_blocked("free", count) is blocked
🎯
همیشه مرزها را تست کن! بیشتر باگ‌ها دقیقاً سرِ مرزها اتفاق می‌افتند (off-by-one). برای هر حد، سه حالت را بسنج: درست زیر، دقیقاً روی، و درست بالای آن.

۶ Mock و ایزوله‌سازی — تست بدون دنیای واقعی

تست‌ها نباید به اینترنت، API واقعی، یا سرویس‌های بیرونی وابسته باشند. اینجاست که «موک» (Mock) وارد می‌شود.

Mock یعنی یک جایگزینِ جعلی برای وابستگی‌های بیرونی. مثلاً وقتی می‌خواهی منطقِ fallback موتور OXYGPT را تست کنی، نمی‌خواهی واقعاً به Gemini وصل شوی (کند است، پول خرج می‌کند، و نتیجه‌اش هر بار فرق می‌کند). به‌جایش یک موک می‌سازی که دقیقاً همان چیزی را برمی‌گرداند که تو تعیین می‌کنی — یا حتی خطا شبیه‌سازی می‌کند.

چرا موک ضروری است؟

🚀

سرعت

بدون تماس شبکه، تست در چند میلی‌ثانیه اجرا می‌شود، نه چند ثانیه.

🎲

قطعیت

پاسخِ جعلی همیشه یکسان است؛ تست flaky (بی‌ثبات) نمی‌شود.

💸

بدون هزینه

هیچ فراخوانی API واقعی و پولی انجام نمی‌شود.

🧨

شبیه‌سازی خطا

می‌توانی خطای ۵۰۳ یا timeout را عمداً بسازی و رفتار بات را بسنجی.

test_market_tool.py 
from unittest.mock import patch, Mock

def test_market_data_uses_cache_on_second_call():
    # Arrange — پاسخِ HTTP جعلی می‌سازیم؛ به شبکه واقعی وصل نمی‌شویم
    fake_response = Mock()
    fake_response.json.return_value = {"data": [...]}

    with patch("tools.requests.get", return_value=fake_response) as mock_get:
        # Act — دو بار صدا می‌زنیم
        get_market_data("EURUSD", "1h")
        get_market_data("EURUSD", "1h")

    # Assert — بار دوم باید از کش بیاید، پس فقط یک تماس شبکه
    assert mock_get.call_count == 1
🧩
انواع جایگزین‌های تست (Test Doubles):

Mock: رفتار جعلی + امکان بررسیِ اینکه چطور صدا زده شد. Stub: فقط یک مقدارِ ثابت برمی‌گرداند. Fake: پیاده‌سازیِ سبک و واقعی (مثل دیتابیسِ درون‌حافظه‌ای). در پایتون معمولاً همه را با unittest.mock می‌سازیم.

تست کد ناهمگام (async)

OXYGPT کاملاً async است (بر پایه asyncio). برای تست توابعِ async از افزونه‌ی pytest-asyncio و برای شبیه‌سازیِ فراخوانی‌هایِ awaitشونده از AsyncMock استفاده می‌کنیم.

test_provider_fallback.py 
import pytest
from unittest.mock import AsyncMock

@pytest.mark.asyncio
async def test_falls_back_to_openai_on_gemini_timeout():
    # Arrange — Gemini با timeout خطا می‌دهد، OpenAI موفق است
    gemini = AsyncMock(side_effect=TimeoutError())
    openai = AsyncMock(return_value="پاسخ از OpenAI")
    engine = Engine(primary=gemini, fallback=openai)

    # Act — یک بار await می‌کنیم
    result = await engine.generate("سلام")

    # Assert — باید به fallback سوییچ کرده باشد
    assert result == "پاسخ از OpenAI"
    openai.assert_awaited_once()
⏱️
هرگز با زمانِ واقعی صبر نکن! برای تستِ منطقِ «بازیابی ۳۰ دقیقه‌ای» ماژول ۵۰۳، asyncio.sleep را موک کن یا زمان را فریز کن — وگرنه تستِ تو ۳۰ دقیقه طول می‌کشد!

۷ ویژگی‌های یک تستِ خوب: اصل FIRST

هر تستی خوب نیست. تست‌های خوب پنج ویژگی دارند که با سرواژه‌ی FIRST شناخته می‌شوند.

F
Fast — سریعباید در میلی‌ثانیه اجرا شود تا مدام اجرایش کنی
I
Isolated — مستقلبه تست دیگر یا ترتیب اجرا وابسته نباشد
R
Repeatable — تکرارپذیرهر بار همان نتیجه؛ نه flaky
S
Self-validating — خودسنجخودش سبز/قرمز می‌گوید؛ بررسی دستی لازم نیست
T
Timely — به‌موقعکنارِ کد (یا قبلش، در TDD) نوشته شود

چند قانون طلایی دیگر

  • یک تست، یک رفتار. اگر تستی چند چیز را می‌سنجد، بشکنش به چند تست.
  • نامِ تست = مشخصات. نامِ تست باید بدون خواندنِ بدنه بگوید چه رفتاری را می‌سنجد؛ مثل test_free_tier_blocks_after_25_requests.
  • تست را ساده نگه دار. در تست از منطقِ پیچیده (حلقه، شرط تودرتو) پرهیز کن؛ تستِ پیچیده خودش باگ می‌گیرد.
  • تکرار در تست اشکالی ندارد. خوانایی مهم‌تر از DRYبودنِ افراطی است.
  • وقتی باگ پیدا شد، اول یک تستِ قرمز بنویس که آن باگ را بازتولید کند، بعد رفعش کن. این‌طور مطمئن می‌شوی باگ دیگر برنمی‌گردد (regression test).
🚫
ضدالگوهایی که باید از آن‌ها دوری کنی:

تستی که هیچ‌وقت قرمز نمی‌شود · تستی که به اینترنت وصل می‌شود · تستی که به تستِ قبلی وابسته است · تستی که چند رفتار را قاطی می‌کند · تستی که فقط برای بالا بردنِ عددِ «پوشش» نوشته شده و چیز معناداری نمی‌سنجد.

۸ تست‌نویسی برای OXYGPT در عمل

حالا همه‌چیز را کنار هم بگذاریم و ببینیم برای پروژه‌ی واقعیِ OXYGPT چه چیزهایی را و چطور تست می‌کنیم.

OXYGPT بخش‌های متنوعی دارد: موتور تولید، لایه‌ی دیتابیس، ابزارها، مدیریت خطای ۵۰۳، و دو ماژول اختیاری. برای هر کدام دسته‌ی متفاوتی از تست معنا دارد:

🗄️ لایه دیتابیس (database.py) — تستِ واحد

با یک دیتابیس موقت (فیکسچرِ temp_db) این‌ها را می‌سنجیم: ساخت پنجره و سقفِ ۵تایی، محدودیتِ ۴۰ پیامِ تاریخچه، شمارنده‌های مصرف، بلاک/آنبلاک، قفل کانال‌های اجباری، و وضعیتِ ۵۰۳ (downgrade/restore).

🧠 موتور تولید (api_http.py) — تستِ یکپارچگی با موک

ارائه‌دهنده‌ها را موک می‌کنیم و می‌سنجیم: چرخشِ round-robin کلیدها، سوییچِ fallback روی خطای قابل‌تکرار، اینکه خطای احراز هویت هرگز fallback را فعال نمی‌کند، و آستانه‌ی خلاصه‌سازی (۵ پیام و ۱۵هزار توکن).

🛠️ ابزارها (tools.py) — تستِ واحد با موکِ شبکه

نرمال‌سازی نماد، کش‌شدنِ داده بازار، هر ۸ سبکِ تولید تصویر، هر تمِ بوکلت HTML، و پاک‌سازیِ HTML (حذف تگ‌های خطرناک مثل script). همه با موکِ HTTP/WebSocket.

🛡️ مدیریت ۵۰۳ (gemini_503_manager.py) — تستِ async با زمانِ فریزشده

اولین ۵۰۳ فقط مدلِ جایگزین می‌دهد؛ دومین ۵۰۳ downgrade سراسری + نوتیفیکیشن + زمان‌بندیِ بازیابی را فعال می‌کند (با موکِ asyncio.sleep). idempotent بودنِ singleton و بازیابیِ مدل‌ها هم سنجیده می‌شود.

📓📡 ماژول‌ها (trade_journal / channel_watcher) — تستِ واحد + یکپارچگی

محاسبات آماری ژورنال (نرخ برد، profit factor) روی داده‌ی نمونه، آستانه‌های نوتیفیکیشن، و در چنل‌واچر: تصمیمِ طبقه‌بندیِ اهمیت (با موکِ classifier)، حذفِ پیامِ تکراری، و مرزِ نگه‌داریِ ۹۰ روزه.

۹ چک‌لیست و جمع‌بندی

هر بار که تست می‌نویسی، این فهرست را مرور کن.

✅ چک‌لیست تستِ حرفه‌ای

  • آیا از الگوی Arrange-Act-Assert پیروی کردم؟
  • آیا هر تست فقط یک رفتار را می‌سنجد؟
  • آیا نامِ تست به‌تنهایی رفتار را توصیف می‌کند؟
  • آیا همه‌ی وابستگی‌های بیرونی (شبکه، API، دیتابیسِ واقعی) را موک کرده‌ام؟
  • آیا تست مستقل است و به ترتیبِ اجرا وابسته نیست؟
  • آیا مرزها را سنجیده‌ام (زیر/روی/بالای حد)؟
  • آیا برای کدِ async از @pytest.mark.asyncio و AsyncMock استفاده کردم؟
  • آیا تست‌ها سریع و بدون sleepِ واقعی‌اند؟
  • آیا مرحله‌ی قرمز را دیدم تا مطمئن شوم تست واقعاً چیزی می‌سنجد؟
