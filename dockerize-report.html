۱ چرا اصلاً داکر؟

تو تا الان احتمالاً بات رو اینطوری اجرا کردی: پایتون نصب، pip install -r requirements.txt، ساختن یک .env، و بعد python telegram.py. این کار روی سیستم خودت جواب می‌ده، ولی وقتی می‌ری روی یک سرور جدید، همین چند خط تبدیل می‌شه به یک عصرِ پر از خطا: نسخه‌ی پایتون فرق داره، یک کتابخانه‌ی سیستمی (مثلِ کتابخانه‌های matplotlib) نیست، و در نهایت جمله‌ی معروف: «رو سیستم من که کار می‌کرد!».

داکر دقیقاً همین درد را حل می‌کند. ایده ساده است:

کد پروژه
telegram.py …
＋
وابستگی‌ها

＋
سیستم‌عامل

Image 📦
همه‌جا یکسان

همه‌چیز (سیستم‌عامل کوچک + پایتون + کتابخانه‌ها + کد تو) داخل یک بسته‌ی واحد به اسم Image بسته‌بندی می‌شه. این بسته روی لپ‌تاپ تو، روی سرور آلمان، و روی سرور رفیقت — دقیقاً یک‌شکل اجرا می‌شه.

✅
مزیت واقعی برای توسعه: دیگه محیط سیستمت رو کثیف نمی‌کنی. می‌خوای پروژه رو پاک کنی؟ یک docker compose down و تمام — هیچ ردی از پایتون و کتابخانه‌ها روی سیستمت نمی‌مونه.

Image در برابر Container

این دو تا رو خیلی‌ها قاطی می‌کنن، ولی تفاوتشون ساده است:

  • Image = نقشه/قالب ثابت (مثل فایل نصب یک برنامه). فقط خونده می‌شه.
  • Container = یک نمونه‌ی در حال اجرا از آن Image (مثل برنامه‌ای که بازش کردی). زنده و پویاست.

از یک Image می‌تونی چند Container بسازی؛ درست مثل اینکه از یک فایل نصب، برنامه رو روی چند کامپیوتر نصب کنی.

۲ کالبدشکافی OXYGPT

قبل از داکرایز کردن هر پروژه‌ای، باید بدونی با چه چیزی طرفی. بیایم ساختار OXYGPT رو بشکافیم:

  • نقطه‌ی ورود: telegram.py — یک پراسسِ تک، بر پایه‌ی asyncio که تا ابد اجرا می‌مونه.
  • موتور: کتابخانه‌ی Telethon برای تلگرام + ارائه‌دهنده‌های هوش مصنوعی (Gemini / OpenAI).
  • وابستگی‌های سنگین: pandas، matplotlib، mplfinance، Pillow — این‌ها برای کامپایل به کتابخانه‌های سیستمی نیاز دارند (نکته‌ی مهم برای Dockerfile).
  • تنظیمات: کاملاً از طریق متغیرهای محیطی (.env) خونده می‌شه — که برای داکر عالیه.

داده‌هایی که باید «زنده» بمانند (State)

این مهم‌ترین بخشِ داکرایز کردنِ یک بات است. اگر این فایل‌ها رو داخل Image نگه داری، با هر بار ری‌بیلد همه‌چیز پاک می‌شه (لاگین تلگرام می‌پره، دیتابیس صفر می‌شه!). پس باید بیرون از Container ذخیره‌شون کنیم:

فایلچیست؟اگر پاک شود…
bot.sessionنشست لاگین تلگرامبات دوباره باید لاگین کند
bot_database.dbدیتابیس اصلی کاربران/تنظیماتتمام کاربران و آمار می‌پرد
trade_journal/journal.dbژورنال معاملاتسوابق کاربر نابود می‌شود
channel_watcher.dbدیتابیس رصد کانالتنظیمات رصد می‌پرد
logs/فایل‌های لاگتاریخچه‌ی خطاها گم می‌شود
🎯
قانون طلایی: کدِ برنامه داخل Image؛ اما داده (State) همیشه بیرون، روی Volume. این جدا کردن، قلبِ داکرایزِ درست است.

۳ مفاهیم حرفه‌ای که فرق را می‌سازند

تو گفتی با مباحث «جدیدتر و حرفه‌ای‌تر» می‌خوای آشنا شی. این چهار مفهوم، جدا کننده‌ی یک Dockerfile آماتور از یک نسخه‌ی Production است.

۱) لایه‌ها و کشِ ساخت (Layer Cache)

هر دستور در Dockerfile یک لایه (Layer) می‌سازد. داکر لایه‌ها را کش می‌کند و اگر ورودی یک لایه تغییر نکرده باشد، دوباره اجرایش نمی‌کند. ترفند حرفه‌ای: اول requirements.txt را کپی و نصب کن، بعد بقیه‌ی کد را. این‌طوری تا وقتی وابستگی‌ها عوض نشدن، با هر تغییر کوچک در کد، نصب مجدد کتابخانه‌ها اتفاق نمی‌افته و ری‌بیلد در چند ثانیه تمام می‌شود.

۲) ساختِ چندمرحله‌ای (Multi-stage Build) 🌟

این ستاره‌ی کار ماست. کتابخانه‌هایی مثل pandas برای نصب به کامپایلر و هدرهای سیستمی (gcc, build-essential) نیاز دارند — حجیم و پرخطر. با Multi-stage، در یک مرحله (builder) این ابزارها را داریم و کتابخانه‌ها را داخل یک virtualenv می‌سازیم؛ سپس در مرحله‌ی نهایی (runtime) فقط همان virtualenv را کپی می‌کنیم — بدون کامپایلر، بدون کش pip.

Stage: builder
gcc + build headers
~900MB
←venv فقط
Stage: runtime
slim + venv
سبک‌تر و امن‌تر

نتیجه: ایمیج کوچک‌تر، سطح حمله‌ی امنیتی کمتر، و دانلود سریع‌تر روی سرور.

۳) کاربر غیرِ root

به‌صورت پیش‌فرض همه‌چیز داخل کانتینر با کاربر root اجرا می‌شه. اگر یک آسیب‌پذیری پیدا بشه، مهاجم root می‌شه. راهکار حرفه‌ای: یک کاربر معمولی (appuser) بساز و برنامه را با آن اجرا کن.

۴) init درست + سیگنال‌ها (tini)

برنامه‌ی ما یک حلقه‌ی asyncio است. اگر پراسسِ پایتون مستقیماً PID 1 باشد، سیگنال‌های خاموش‌سازی (SIGTERM هنگام docker stop) درست مدیریت نمی‌شوند و ممکن است تسک‌ها یتیم بمانند. یک init سبک به اسم tini این را حل می‌کند و خاموش‌سازیِ تمیز می‌دهد.

🧩
و در نهایت Volume: پوشه‌ای روی هاست که به داخل کانتینر «وصل» می‌شود. هر چیزی آنجا نوشته شود، بیرون از کانتینر باقی می‌ماند — دقیقاً همان جایی که State بات را نگه می‌داریم.

۴ نوشتن Dockerfile — خط به خط

این همان Dockerfileی است که برای OXYGPT ساختیم. هر بخش را با توضیح فارسی می‌آوریم.

مرحله‌ی ۱: builder (ساخت وابستگی‌ها)

Dockerfileکپی
# از یک پایتونِ سبک به‌عنوان پایه‌ی مرحله‌ی builder استفاده می‌کنیم
FROM python:3.12-slim AS builder

# رفتار pip/python را در زمان build سریع و بی‌سروصدا می‌کنیم
ENV PYTHONDONTWRITEBYTECODE=1 PYTHONUNBUFFERED=1 PIP_NO_CACHE_DIR=1

# ابزار کامپایل که فقط اینجا لازم است (به ایمیج نهایی نمی‌رسد)
RUN apt-get update && apt-get install -y --no-install-recommends \
        build-essential gcc libjpeg-dev zlib1g-dev libfreetype6-dev pkg-config \
    && rm -rf /var/lib/apt/lists/*

# یک virtualenv مستقل می‌سازیم تا بعداً کامل کپی‌اش کنیم
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

WORKDIR /app
# فقط requirements را اول کپی می‌کنیم → کشِ لایه (rebuild سریع)
COPY requirements.txt .
RUN pip install --upgrade pip && pip install -r requirements.txt

مرحله‌ی ۲: runtime (ایمیج نهاییِ سبک)

Dockerfile (ادامه)کپی
FROM python:3.12-slim AS runtime
ENV PATH="/opt/venv/bin:$PATH" PYTHONUNBUFFERED=1 TZ=UTC

# فقط کتابخانه‌های سیستمیِ زمان اجرا (بدون کامپایلر)
RUN apt-get update && apt-get install -y --no-install-recommends \
        libjpeg62-turbo libfreetype6 libpng16-16 tini curl \
    && rm -rf /var/lib/apt/lists/*

# venvِ آماده را از مرحله‌ی builder کپی می‌کنیم — دیگر pip install نداریم
COPY --from=builder /opt/venv /opt/venv

WORKDIR /app
# کاربر غیرِ root برای امنیت
RUN useradd --create-home --uid 10001 appuser
COPY --chown=appuser:appuser . .
RUN mkdir -p /app/logs /app/data && chown -R appuser:appuser /app/logs /app/data
USER appuser

# هلث‌چک: چون پورت HTTP نداریم، فقط سالم بودن importها را می‌سنجیم
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD python -c "import telethon, google.generativeai" || exit 1

# tini = init سبک برای سیگنال‌ها؛ entrypoint داده‌ها را symlink می‌کند
ENTRYPOINT ["tini", "--", "/app/deploy/entrypoint.sh"]
CMD ["python", "telegram.py"]
💡
دقت کن که هیچ رمز و کلیدی داخل Dockerfile نیست. تمام secretها در زمان اجرا از .env تزریق می‌شوند — هرگز داخل ایمیج «پخته» نمی‌شوند.

۵ فایل .dockerignore

این فایل شبیه .gitignore است ولی برای داکر. هر چیزی که اینجا فهرست کنی، اصلاً به «کانتکستِ ساخت» فرستاده نمی‌شود. سه فایده دارد:

  • سرعت: فایل‌های حجیم و بی‌ربط ارسال نمی‌شوند → build سریع‌تر.
  • امنیت: جلوی نشتِ .env و *.session به داخل ایمیج را می‌گیرد.
  • تمیزی: کش، لاگ و دیتابیس‌های محلی وارد ایمیج نمی‌شوند.
.dockerignoreکپی
# کنترل نسخه
.git
.github
# کش‌های پایتون
__pycache__/
*.py[cod]
# ⚠️ رازها و نشست‌ها — هرگز داخل ایمیج نروند
.env
*.session
*.session-journal
# دیتابیس‌ها (روی Volume نگه داشته می‌شوند)
*.db
*.db-shm
*.db-wal
# داده‌های تولیدشده در زمان اجرا
logs/
*.log
*.png
market_data_cache.json
# خودِ فایل‌های داکر و مستندات لازم نیستند
Dockerfile
docker-compose.yml
*.md
deploy/*
# …اما entrypoint باید داخل ایمیج باشد
!deploy/entrypoint.sh
🔐
مهم‌ترین خطِ امنیتی: اگر .env و *.session را ignore نکنی و ایمیج را جایی پابلیک کنی، عملاً توکن بات و لاگین تلگرامت لو می‌رود. این خط شوخی ندارد.

۶ Docker Compose و راز ماندگاری داده

می‌شد بات را با یک دستور طولانیِ docker run … اجرا کرد، ولی حرفه‌ای‌ها از Compose استفاده می‌کنند: کل پیکربندی در یک فایل خوانا، و اجرا فقط با docker compose up -d.

docker-compose.ymlکپی
services:
  oxygpt:
    build: .              # از Dockerfile همین پوشه بساز
    image: oxygpt:latest
    container_name: oxygpt
    restart: unless-stopped # اگر کرش کرد یا سرور ری‌استارت شد، خودکار بالا بیاد

    env_file:
      - .env               # همه‌ی secretها از اینجا

    volumes:
      - ./data:/app/data   # ★ کل State اینجا زندگی می‌کند و پایدار است

    deploy:                # سقف مصرف رم تا یک تسکِ خراب سرور را نخورد
      resources: {limits: {memory: 1g}}

    logging:               # چرخش لاگ تا دیسک پر نشود
      driver: json-file
      options: {max-size: "10m", max-file: "3"}

مشکل ظریفِ ماندگاری در این پروژه

اینجا یک نکته‌ی حرفه‌ای هست که خیلی‌ها گیرش می‌افتند: OXYGPT فایل‌های State را در مسیرهای پراکنده می‌سازد (bot_database.db در ریشه، trade_journal/journal.db در زیرپوشه، و…). mount کردنِ تک‌تکِ این فایل‌ها شکننده است. راه‌حلِ تمیز: یک پوشه‌ی /app/data را mount می‌کنیم و یک اسکریپت entrypoint، هر فایل State را با symlink به داخل همان پوشه هدایت می‌کند — بدون اینکه حتی یک خط از کد پایتون تغییر کند.

deploy/entrypoint.shکپی
#!/usr/bin/env sh
set -e
DATA_DIR="/app/data"
mkdir -p "$DATA_DIR" "$DATA_DIR/logs"

# هر فایل State را به داخل پوشه‌ی پایدار symlink می‌کند
link() {
  real="$1"; target="$DATA_DIR/$2"
  mkdir -p "$(dirname "$real")"
  # اگر ایمیج یک فایل واقعی داشت و volume خالی بود، یک‌بار منتقلش کن
  [ -e "$real" ] && [ ! -L "$real" ] && [ ! -e "$target" ] && mv "$real" "$target"
  [ -e "$target" ] || : > "$target"
  rm -f "$real"; ln -s "$target" "$real"
}

link /app/bot.session                     bot.session
link /app/bot_database.db                 bot_database.db
link /app/channel_watcher_user.session    channel_watcher_user.session
link /app/channel_watcher/channel_watcher.db channel_watcher.db
link /app/trade_journal/journal.db        journal.db
rm -rf /app/logs; ln -s "$DATA_DIR/logs" /app/logs

exec "$@"   # تحویل به دستور اصلی (python telegram.py)
🧠
چرا symlink؟ چون بدون دست‌زدن به سورس پایتون، تمام نوشتن‌ها به فایل‌های State عملاً روی ./dataی هاست می‌افتد. پروژه را حذف کن، دوباره بساز — لاگین و دیتابیس‌ها سرِ جایشان‌اند. این همان چیزی است که «داکرایزِ درست» را از «فقط کار می‌کند» جدا می‌کند.

۷ نصب‌کننده‌ی تک‌خطی (بخش سرگرمی 🎉)

حالا می‌رسیم به همان چیزی که عاشقش شدی: تجربه‌ی نصبِ تک‌خطی. این دقیقاً همان الگویی است که پروژه‌ی خودت دارد. کاربر فقط این را می‌زند و بقیه‌اش جادوست:

terminalکپی
bash <(curl -fsSL https://raw.githubusercontent.com/nauseaTT/Telegram-Robatsaz/main/deploy/install.sh)

این یک خط دقیقاً چه کار می‌کند؟

  • curl -fsSL <url> محتوای اسکریپت را از گیت‌هاب دانلود می‌کند.
  • <( … ) که به آن Process Substitution می‌گویند، خروجی curl را مثل یک فایل موقت به bash می‌دهد.
  • پس bash آن اسکریپت را بی‌آنکه چیزی روی دیسک بماند، اجرا می‌کند.
⚠️
هشدار امنیتی که همیشه باید بدانی: اجرای curl | bash یعنی به آن ریپازیتوری اعتماد کامل داری، چون هر کدی داخل اسکریپت باشد روی سیستمت اجرا می‌شود. برای پروژه‌ی خودت عالیه؛ برای منابع ناشناس، اول اسکریپت را بخوان.

اسکریپت ما این مراحل را طی می‌کند

بررسی پیش‌نیازها

وجود Docker و Docker Compose را چک می‌کند و در صورت نبود، روی لینوکس پیشنهاد نصب می‌دهد.

گرفتن سورس

ریپو را در ~/oxygpt کلون می‌کند؛ اگر از قبل بود، فقط آپدیتش می‌کند (idempotent).

پرسیدن کلیدها

به‌صورت تعاملی API_ID، API_HASH، توکن بات و کلید Gemini را می‌پرسد و در .env می‌نویسد.

ساخت و اجرا

docker compose build و سپس up -d — بات در پس‌زمینه بالا می‌آید.

راهنمای پایانی

دستورهای مدیریتی (لاگ، ری‌استارت، توقف) را با رنگ و لعاب چاپ می‌کند.

تکه‌ی کلیدیِ اسکریپت

deploy/install.shکپی
#!/usr/bin/env bash
set -euo pipefail    # در اولین خطا متوقف شو (اسکریپتِ ایمن)

REPO_URL="https://github.com/nauseaTT/Telegram-Robatsaz.git"
INSTALL_DIR="${OXYGPT_DIR:-$HOME/oxygpt}"

# ۱) داکر هست؟ اگر نه، روی لینوکس پیشنهاد نصب بده
command -v docker >/dev/null || curl -fsSL https://get.docker.com | sh

# ۲) کلون یا آپدیت
if [ -d "$INSTALL_DIR/.git" ]; then
  git -C "$INSTALL_DIR" pull --ff-only
else
  git clone --depth 1 "$REPO_URL" "$INSTALL_DIR"
fi
cd "$INSTALL_DIR"

# ۳) اگر .env نبود، کلیدها را بپرس (تابع ask در اسکریپت کامل)
[ -f .env ] || { cp .env.example .env; ask TELEGRAM_BOT_TOKEN "Bot Token" 1; }

# ۴) بساز و اجرا کن
mkdir -p data data/logs
docker compose up -d --build

echo "✔ OXYGPT بالا آمد! لاگ‌ها: docker compose logs -f"
✨
نسخه‌ی کاملِ این اسکریپت (با بنر ASCII، رنگ‌ها، و پرسش‌های تعاملیِ همه‌ی کلیدها) داخل فایل deploy/install.sh پروژه قرار دارد و آماده‌ی استفاده است.

۸ اجرا و مدیریت روزمره

اگر نمی‌خواهی از نصب‌کننده استفاده کنی، دستی هم ساده است. داخل پوشه‌ی پروژه:

دستورهای پرکاربردکپی
# ۱) کلیدها را در .env بگذار
cp .env.example .env && nano .env

# ۲) بساز و در پس‌زمینه بالا بیاور
docker compose up -d --build

# دیدن لاگ زنده
docker compose logs -f

# وضعیت کانتینر
docker compose ps

# ری‌استارت بعد از تغییر کد
docker compose up -d --build

# توقف و حذف کانتینر (داده‌ها در ./data می‌مانند)
docker compose down
می‌خواهی…دستور
بات را روشن کنیdocker compose up -d
لاگ‌ها را ببینیdocker compose logs -f
بعد از تغییر کد آپدیت کنیdocker compose up -d --build
بات را خاموش کنیdocker compose down
از دیتابیس بکاپ بگیریtar czf backup.tgz data/
💾
بکاپ‌گیری راحت شد: چون کل State در ./data است، فقط از همین یک پوشه بکاپ بگیر و روی سرور جدید بازش کن — بات با همان لاگین و دیتابیس‌ها بالا می‌آید.

۹ چک‌لیست نهاییِ داکرایز حرفه‌ای

قبل از اینکه بگویی «تمام شد»، این‌ها را تیک بزن:

موردوضعیت در پروژه‌ی ما
ساخت چندمرحله‌ای (سبک‌سازی ایمیج)✔ انجام شد
کش لایه با کپی زودهنگام requirements✔ انجام شد
اجرا با کاربر غیرِ root✔ appuser
init درست برای سیگنال‌ها (tini)✔ انجام شد
secretها بیرون از ایمیج (env_file)✔ انجام شد
ماندگاری داده روی Volume✔ ./data + symlink
.dockerignore برای امنیت و سرعت✔ انجام شد
restart خودکار + سقف رم + چرخش لاگ✔ انجام شد
هلث‌چک✔ انجام شد
نصب تک‌خطی✔ deploy/install.sh
🚀
قدم‌های بعدیِ حرفه‌ای‌تر (اختیاری): پوش کردن ایمیج به یک Registry (مثل GitHub Container Registry)، اسکنِ امنیتی با docker scout، و ساخت خودکار با GitHub Actions در هر push.
